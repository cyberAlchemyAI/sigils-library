# Repository Observability Package

The repository observability package is the local telemetry home for sigils used in any consuming repository.

Arcanum defines schemas, hooks, and lifecycle rules. The consuming repository owns the telemetry generated by its own sigil usage. This keeps the framework portable and lets every repository decide how much signal to retain.

## Default Location

Use this path by default:

```text
.arcanum/observability/
```

This keeps sigil telemetry separate from product code, docs, and the Arcanum source.

Arcanum sigil invocation telemetry remains under `.arcanum/observability/signals/` by default.

When Arcanum is installed through bootstrap, `.arcanum/necronomicon/OBSERVABILITY.md` is the runtime handoff contract that points adapters to this package. The observability package stores telemetry; Necronomicon decides how runtime invocations should hand telemetry to it.

## Default Layout

```text
.arcanum/observability/
  README.md
  config.json
  reflection-state.json
  signals/
    sigil-invocations.jsonl
  by-sigil/
    .gitkeep
  by-capability/
    .gitkeep

      .gitkeep

    .gitkeep

    .gitkeep
```

## Storage Options

### Option 1: Central Ledger

All sigil invocations append to one JSONL file:

```text
.arcanum/observability/signals/sigil-invocations.jsonl
```

Pros:

- simplest hook implementation,
- preserves chronological order across all sigils,
- easy to count global thresholds,
- one file to archive, inspect, or ship.

Cons:

- single file can grow noisy,
- per-sigil reflection requires filtering,
- merge conflicts are more likely if many agents append concurrently.

Use when the repository has low to moderate sigil usage or wants the simplest portable setup.

### Option 2: Rebuildable Lookup Indexes

Each central ledger row can be indexed by sigil or capability:

```text
.arcanum/observability/by-sigil/<sigil-name>.jsonl
.arcanum/observability/by-capability/<kind>/<capability-id>.jsonl
```

Pros:

- easy to reflect on one sigil,
- lower noise per file,
- clearer ownership when sigils have different thresholds,
- indexes can be deleted and rebuilt from the central ledger.

Cons:

- index rows are not full telemetry rows,
- consumers must follow the `ledger` and `line` reference to retrieve the canonical event.

Use when individual sigils or capabilities are high-volume, high-risk, or owned by different maintainers.

### Option 3: Mirrored Ledgers

Write every invocation to the central ledger and also mirror full telemetry rows into per-sigil files:

```text
.arcanum/observability/signals/sigil-invocations.jsonl
.arcanum/observability/by-sigil/<sigil-name>.jsonl
```

Pros:

- preserves global chronology,
- supports focused per-sigil reflection,
- works well with thresholds at both repository and sigil levels,
- easiest to evolve from simple usage to serious observability.

Cons:

- duplicate storage if events are mirrored,
- requires clear rule for whether per-sigil files are source-of-truth or derived,
- slightly more setup.

This is no longer the recommended default because it duplicates telemetry and complicates dedupe. Use it only for a consuming repo that explicitly wants independent per-sigil archival files.

## Recommended Default

Use the central ledger with rebuildable reference indexes:

```json
{
  "storage_model": "central-ledger-reference-indexes",
  "source_of_truth": "signals/sigil-invocations.jsonl",
  "per_sigil_index_path": "by-sigil/<sigil-name>.jsonl",
  "per_capability_index_path": "by-capability/<kind>/<capability-id>.jsonl"
}
```

This gives one reliable append path, compact per-artifact lookup, and a clean rebuild story.

## Config File

`config.json` should define:

```json
{
  "version": "0.2.0",
  "storage_model": "central-ledger-reference-indexes",
  "source_of_truth": "signals/sigil-invocations.jsonl",
  "per_sigil_index_path": "by-sigil/<sigil-name>.jsonl",
  "per_capability_index_path": "by-capability/<kind>/<capability-id>.jsonl",
  "reflection_path": "reflections/",
  "thresholds": {
    "meaningful_executions": 5,
    "generated_outputs": 10,
    "related_workflow_gaps": 3,
    "severe_workflow_gaps": 1
  }
}
```

## Reflection State

`reflection-state.json` tracks counters since the last reflection:

```json
{
  "version": "0.1.0",
  "last_reflection_at": null,
  "counters": {
    "meaningful_executions": 0,
    "generated_outputs": 0,
    "related_workflow_gaps": 0,
    "severe_workflow_gaps": 0,
    "quality_bar_failures": 0,
    "output_contract_drift_events": 0
  },
  "by_sigil": {},
  "by_capability": {}
}
```

## Hook Integration

Use [Sigil Observability Hook](SIGIL-OBSERVABILITY-HOOK.md) after each meaningful sigil execution. The hook appends one invocation event to the configured ledger and reports whether reflection is required.

Background extraction, observation, append, and hook-health work must write operational audit rows to `hooks/hook-operations.jsonl`, not to the capability telemetry ledger. See [Hook Operations Ledger](HOOK-OPERATIONS-LEDGER.md).

Long-running sigil and spell executions should use observed run bundles under `runs/`. See [Observed Runs](OBSERVED-RUNS.md).

Use `scripts/rebuild-observability-indexes.sh` after manual ledger edits or imports. Use `scripts/compact-observability-store.sh` to dedupe the central ledger and remove superseded hook envelope copies.
